Purpose
This document breaks the debian_dev branch into feature-sized units for later backporting into the Termux-based main branch. The goal is not to merge debian_dev as-is. The goal is to decide, feature by feature, what can be implemented in the Android/Termux home-server project and how it should be adapted.

Validation and environment notes from the current Termux host
- nginx is installed and built with --with-http_auth_request_module, so nginx auth_request is available here.
- Installed and available in the current Termux environment: nginx, node, npm, sshd, filebrowser, ttyd, python3, ss, df, ifconfig, termux-wake-lock.
- Not available right now: busybox, pyftpdlib.
- These Linux-oriented commands do not work in the current Termux environment: df -T, df -B1, hostname -I, ip route get 1.1.1.1.
- ifconfig works and exposes wlan0 with the LAN IP, so Android-safe host detection should use ifconfig or another Termux-safe network probe instead of Debian-only commands.

[x] Feature 01: Hardened Termux startup flow in start.sh
Feature description
debian_dev rewrites start.sh to be stricter and safer. It adds set -euo pipefail, environment-driven paths, conditional checks for installed commands, repo-local nginx startup, localhost-only bindings for filebrowser and ttyd, and better cleanup logic.
Debian branch tech used
- Bash strict mode and helper functions.
- HOME_SERVER_DIR and FILEBROWSER_ROOT environment variables.
- pkill -f based cleanup for node, next, nginx, filebrowser, ttyd.
- filebrowser with -a 127.0.0.1 -b /files --noauth.
- ttyd with -W -i 127.0.0.1.
- nginx -p <repo> -c <repo>/nginx.conf.
- Dynamic host IP detection using ip route get and hostname -I.
How to implement in main Termux Android environment
- Port the strict startup structure, env variables, repo-local nginx startup, conditional command checks, and localhost-only bindings.
- Do not copy the host-IP detection logic as-is because ip route get and hostname -I fail in this Termux environment.
- Replace host-IP detection with a Termux-safe method, for example parsing ifconfig wlan0, or fall back to showing localhost plus a note for LAN access.
- Keep termux-wake-lock because it works here and is valuable on Android.
Compatibility
Compatible with adaptation required for host-IP detection.

[x] Feature 09: Monitoring improvements and new telemetry endpoints
Feature description
debian_dev replaces the old loadavg-based CPU percentage with a real CPU snapshot calculation and adds endpoints for connected users, storage inventory, debug logs, and verbose logging control.
Debian branch tech used
- os.cpus snapshot delta calculation for CPU usage
- ss -tn state established for connection data
- df -B1 -T -x tmpfs -x devtmpfs for storage
- In-memory debug event buffer and markdown log output
- /api/connections, /api/storage, /api/logs, /api/logging
How to implement in main Termux Android environment
- Port the CPU snapshot logic directly. That is a clean improvement over the old loadavg approximation.
- Port the connection and debug-log endpoints with minimal changes because ss works in the current environment.
- Rewrite the storage endpoint for Android. The current Termux host does not support df -T or df -B1, so the Debian storage logic is not portable as-is.
- For Android storage, use df -kP plus mount parsing and Android-specific mount filters such as /storage, /mnt/user/0/emulated, fuse, and f2fs.
Compatibility
Compatible for CPU, connections, and logs. Storage needs Android-specific rework.
Implementation note on this Termux host
- Node returns an empty os.cpus() array because CPU times are not exposed through /proc here, so main uses a top-based CPU fallback on Android instead of the Debian snapshot path.
- ss and /proc/net/tcp are permission-blocked in this environment, so /api/connections uses recent authenticated request tracking with trust-proxy support rather than low-level socket enumeration.
- /api/storage returns deduplicated user-relevant Android storage pools and avoids double-counting the shared-storage fuse view alongside /data.

Conclusion
Most of the important debian_dev work is portable to main, but it should be split into Android-safe backports rather than merged wholesale. The strongest direct imports are the auth model, nginx gateway improvements, service-control hardening, env configuration, and removal of hardcoded host assumptions. The biggest area that needs custom Android work is storage inspection, plus any host-IP detection that currently assumes standard Linux networking commands.
